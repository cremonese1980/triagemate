### 11.1.a — Enforce JSON Logging Everywhere

#### Goal

Ensure all logs emitted in production-like environments are structured JSON logs.

Human-readable logs remain acceptable only for local development profiles.

---

#### Technical Implementation

**Step 1: Add JSON encoder dependency**

Add to `pom.xml`:

```xml
<dependency>
    <groupId>net.logstash.logback</groupId>
    <artifactId>logstash-logback-encoder</artifactId>
    <version>7.4</version>
</dependency>
```

**Why this encoder:**
- Native MDC support (automatic inclusion of MDC fields)
- Structured providers architecture (timestamp, level, logger, message, mdc, context)
- Battle-tested in production environments
- ELK/Grafana Loki compatible output

---

**Step 2: Configure Logback for JSON output**

Create or update `src/main/resources/logback-spring.xml`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>

    <!-- 
        JSON Console Appender for production-like profiles 
        
        PROFILE BEHAVIOR:
        - <springProfile name="docker,prod"> means: docker OR prod (either active)
        - <springProfile name="!docker,!prod"> means: NOT docker AND NOT prod (neither active)
        
        This asymmetry is Logback-specific and counterintuitive.
        Result: JSON logging active when docker=true OR prod=true
                Plain text logging active when docker=false AND prod=false
    -->
    <springProfile name="docker,prod">
        <appender name="JSON_CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
            <encoder class="net.logstash.logback.encoder.LoggingEventCompositeJsonEncoder">
                <providers>
                    <!-- ISO-8601 timestamp -->
                    <timestamp>
                        <fieldName>@timestamp</fieldName>
                        <timeZone>UTC</timeZone>
                    </timestamp>

                    <!-- Log level -->
                    <logLevel>
                        <fieldName>level</fieldName>
                    </logLevel>

                    <!-- Logger name (full class path) -->
                    <loggerName>
                        <fieldName>logger</fieldName>
                    </loggerName>

                    <!-- Thread name -->
                    <threadName>
                        <fieldName>thread</fieldName>
                    </threadName>

                    <!-- Log message -->
                    <message>
                        <fieldName>message</fieldName>
                    </message>

                    <!-- MDC fields (requestId, correlationId, eventId, etc.) -->
                    <mdc/>

                    <!-- Application context (spring.application.name) -->
                    <context>
                        <fieldName>service</fieldName>
                    </context>

                    <!-- Stack trace for exceptions -->
                    <stackTrace>
                        <fieldName>stack_trace</fieldName>
                    </stackTrace>

                </providers>
            </encoder>
        </appender>

        <root level="INFO">
            <appender-ref ref="JSON_CONSOLE"/>
        </root>
    </springProfile>

    <!-- Plain text console for local development -->
    <springProfile name="!docker,!prod">
        <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
            <encoder>
                <pattern>%d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
            </encoder>
        </appender>

        <root level="DEBUG">
            <appender-ref ref="CONSOLE"/>
        </root>
    </springProfile>

</configuration>
```

**Key design decisions:**
- `<springProfile>` discriminates between dev and prod logging
- UTC timezone for `@timestamp` (cross-region correlation)
- `mdc` provider automatically includes all MDC fields
- `context` provider exposes `spring.application.name` as `service` field
- Stack traces included only when exceptions present

### 11.1.b — Mandatory Fields Policy

#### Goal

Ensure every operationally relevant log line contains required trace fields.

This enables full traceability across the event pipeline.

---

#### Required Fields

**Universal fields (all business logs):**

```
requestId       - Unique request identifier
correlationId   - Cross-service correlation identifier
service         - Service name (from spring.application.name)
```

**Event-based processing:**

```
eventId         - Event identifier (when processing Kafka messages)
```

**Decision processing:**

```
decisionOutcome - Decision result (ACCEPTED, REJECTED, DUPLICATE, INVALID)
```

---

#### Technical Implementation

**Step 1: Populate MDC at processing boundaries**

**Location:** `InputReceivedConsumer.java` (or wherever Kafka messages enter)

```java
@KafkaListener(topics = "${triagemate.topics.input-received}")
public void consume(ConsumerRecord<String, String> record) {
    try {
        // Parse event envelope
        EventEnvelope envelope = parseEnvelope(record.value());
        
        // Populate MDC with trace fields
        MDC.put("requestId", envelope.trace().requestId());
        MDC.put("correlationId", envelope.trace().correlationId());
        MDC.put("eventId", envelope.eventId());
        
        // Business logic
        processEvent(envelope);
        
    } finally {
        // CRITICAL: Clear MDC to prevent thread pool pollution
        MDC.clear();
    }
}
```

**Why `finally` block:**
- Kafka listener uses thread pool
- Without clearing, next message inherits previous MDC
- Causes trace ID pollution and false correlation

---

**Step 2: Add decision outcome to logs**

**Location:** `DecisionService.java` (or wherever decisions are made)

```java
public DecisionResult executeDecision(InputReceivedEvent event) {
    // Execute decision logic
    DecisionResult result = // ... decision logic
    
    // Add outcome to MDC for subsequent logs
    MDC.put("decisionOutcome", result.outcome().name());
    
    // Log decision with structured outcome field
    log.info("decision executed outcome={} reason={}", 
             result.outcome(), 
             result.reason());
    
    return result;
}
```

**Why structured logging:**
- `outcome={}` creates structured field in JSON
- Enables metric extraction from logs
- Supports Grafana Loki queries: `{service="triagemate-triage"} | json | outcome="REJECTED"`

### 11.1.c — Log Level Governance

#### Goal

Define and enforce a clear logging level policy to maintain a useful signal-to-noise ratio.

Reduce log volume without losing critical operational visibility.

---

#### Logging Level Policy

| Level   | Usage                              | Examples                                      |
|---------|------------------------------------|-----------------------------------------------|
| ERROR   | Unrecoverable failures             | Kafka publish failure, database constraint violation |
| WARN    | Degraded but recoverable behaviour | Retry on transient failure, duplicate event detected |
| INFO    | Business state transitions         | Decision executed, event accepted, outbox published |
| DEBUG   | Diagnostic information             | Deserialization details, transaction boundaries |

---

#### Technical Implementation

**Step 1: Audit existing log statements**

**Search command:**

```bash
grep -rn "log\." src/main/java/com/triagemate/triage/ --include="*.java"
```

**Review checklist for each log:**
- Is this a business state transition? → INFO
- Is this a recoverable error? → WARN
- Is this an unrecoverable failure? → ERROR
- Is this diagnostic detail? → DEBUG

---

**Step 2: Remove noisy INFO logs**

**Anti-patterns to eliminate:**

```java
// BAD: Polling loop noise
log.info("polling outbox for pending messages");

// GOOD: Log only when action taken
if (!pendingMessages.isEmpty()) {
    log.info("publishing {} pending messages", pendingMessages.size());
}
```

```java
// BAD: Framework noise
log.info("transaction started");
log.info("transaction committed");

// GOOD: Remove or move to DEBUG
log.debug("transaction committed for eventId={}", eventId);
```

```java
// BAD: Repetitive technical detail
log.info("deserializing JSON payload");

// GOOD: Log only on failure
try {
    Event event = deserialize(json);
} catch (JsonException e) {
    log.error("deserialization failed", e);
    throw e;
}
```

---

**Step 3: Standardize business state logs**

**Standard business log statements:**

```java
// Decision executed (INFO)
log.info("decision executed outcome={} reason={}", outcome, reason);

// Event accepted (INFO)
log.info("event accepted eventType={}", event.type());

// Event rejected (WARN - not an error, but noteworthy)
log.warn("event rejected reason={}", rejection.reason());

// Duplicate detected (WARN - idempotency working as designed)
log.warn("duplicate event detected eventId={}", eventId);

// Outbox published (INFO)
log.info("outbox event published topic={} partition={}", topic, partition);
```

**Pattern:**
- One log statement per operation
- Structured fields for queryability
- Consistent verb tense (past tense: "executed", "accepted", "published")

---

**Step 4: Avoid unnecessary stack traces**

**Anti-pattern:**

```java
log.error("processing failed", e);
```

**When to include stack traces:**
- ERROR level only
- Unexpected exceptions (not business validation failures)
- Root cause unknown

**When to omit stack traces:**
- Expected validation failures (log as WARN without exception)
- Business rule violations (log outcome, not stack trace)
- Duplicate detection (expected behavior, not an error)

**Correct patterns:**

```java
// Business validation failure - no stack trace needed
if (!validator.validate(event)) {
    log.warn("validation failed eventId={} errors={}", 
             event.eventId(), 
             validator.errors());
    return ValidationResult.invalid();
}

// Unexpected technical failure - include stack trace
try {
    repository.save(entity);
} catch (DataAccessException e) {
    log.error("database persistence failed eventId={}", eventId, e);
    throw e;
}
```

### 11.1.d — Sensitive Data Policy

#### Goal

Prevent leakage of personally identifiable information (PII) and secrets in logs.

Ensure GDPR/HIPAA compliance and operational security.

---

#### Sensitive Data Categories

**Category 1: Patient identifiers**
- Patient ID (e.g., `PAT-12345`)
- Medical record numbers
- Device serial numbers linked to patients

**Category 2: Authentication material**
- API keys
- Bearer tokens
- Database passwords

**Category 3: Protected health information (PHI)**
- Diagnosis codes
- Treatment details
- Clinical observations

**Step 2: Implement field masking utility**

**Create:** `com.triagemate.triage.logging.SensitiveDataMasker`

```java
import java.nio.charset.StandardCharsets;
import java.security.MessageDigest;
import java.security.NoSuchAlgorithmException;

public class SensitiveDataMasker {
    
    /**
     * Masks patient ID with deterministic hash for correlation.
     * 
     * Example: PAT-12345 -> PAT-a3f2e1b4
     * 
     * Uses SHA-256 truncated to 8 chars for:
     * - Deterministic: same patientId always produces same hash
     * - Collision-resistant: 8-char hex = 4 billion combinations
     * - Non-reversible: cannot recover original from hash
     */
    public static String maskPatientId(String patientId) {
        if (patientId == null || patientId.isEmpty()) {
            return "***";
        }
        
        String prefix = patientId.contains("-") 
            ? patientId.substring(0, patientId.indexOf('-') + 1)
            : "PAT-";
        
        String hash = sha256Truncated(patientId, 8);
        return prefix + hash;
    }
    
    /**
     * Masks device serial number with deterministic hash.
     * 
     * Example: DEV-SN-987654321 -> DEV-SN-7b3e9f2a
     */
    public static String maskDeviceSerial(String serialNo) {
        if (serialNo == null || serialNo.isEmpty()) {
            return "***";
        }
        
        String prefix = serialNo.contains("-") 
            ? serialNo.substring(0, serialNo.lastIndexOf('-') + 1)
            : "DEV-SN-";
        
        String hash = sha256Truncated(serialNo, 8);
        return prefix + hash;
    }
    
    /**
     * Redacts authentication tokens completely.
     */
    public static String redactToken(String token) {
        return "[REDACTED]";
    }
    
    /**
     * Generate truncated SHA-256 hash (hex encoded).
     * 
     * @param input Input string
     * @param length Number of hex characters to return
     * @return Truncated hex hash
     */
    private static String sha256Truncated(String input, int length) {
        try {
            MessageDigest digest = MessageDigest.getInstance("SHA-256");
            byte[] hash = digest.digest(input.getBytes(StandardCharsets.UTF_8));
            
            // Convert to hex and truncate
            StringBuilder hex = new StringBuilder();
            for (int i = 0; i < Math.min(hash.length, length / 2); i++) {
                String h = Integer.toHexString(0xff & hash[i]);
                if (h.length() == 1) hex.append('0');
                hex.append(h);
            }
            
            return hex.toString().substring(0, length);
            
        } catch (NoSuchAlgorithmException e) {
            // SHA-256 should always be available
            throw new RuntimeException("SHA-256 not available", e);
        }
    }
}
```

**Why deterministic hash:**
- Same patient always produces same masked ID (correlation works)
- No collision risk with suffix-based masking (PAT-***45 would collide)
- 8-char hex = 2^32 combinations (4 billion) - sufficient for healthcare scale
- Non-reversible - cannot recover original patient ID from hash
- Stable across service restarts (same input → same hash)

**Step 4: Add Logback masking filter (defense-in-depth)**

**Create:** `src/main/java/com/triagemate/triage/logging/SensitiveDataFilter.java`

```java
import ch.qos.logback.classic.spi.ILoggingEvent;
import ch.qos.logback.classic.spi.LoggingEvent;
import ch.qos.logback.core.filter.Filter;
import ch.qos.logback.core.spi.FilterReply;

import java.util.HashMap;
import java.util.Map;
import java.util.regex.Matcher;
import java.util.regex.Pattern;

/**
 * Rewrites log events to mask sensitive data.
 * 
 * CRITICAL: Does NOT deny events (would lose ERROR logs).
 * Instead, rewrites message and MDC fields in-place.
 */
public class SensitiveDataFilter extends Filter<ILoggingEvent> {
    
    private static final Pattern PATIENT_ID_PATTERN = 
        Pattern.compile("PAT-\\d{5}");
    
    private static final Pattern DEVICE_SERIAL_PATTERN = 
        Pattern.compile("DEV-SN-\\d{9}");
    
    @Override
    public FilterReply decide(ILoggingEvent event) {
        // Rewrite message
        String originalMessage = event.getFormattedMessage();
        String maskedMessage = maskSensitiveData(originalMessage);
        
        // Rewrite MDC fields
        Map<String, String> originalMdc = event.getMDCPropertyMap();
        Map<String, String> maskedMdc = maskMdcFields(originalMdc);
        
        // If changes detected, log security violation
        boolean messageChanged = !originalMessage.equals(maskedMessage);
        boolean mdcChanged = !originalMdc.equals(maskedMdc);
        
        if (messageChanged || mdcChanged) {
            System.err.println("SECURITY: Unmasked sensitive data detected and auto-masked in log");
            
            // Create new event with masked data
            LoggingEvent maskedEvent = new LoggingEvent(
                event.getLoggerName(),
                event.getLoggerContextVO(),
                event.getLevel(),
                maskedMessage,
                event.getThrowableProxy(),
                event.getArgumentArray()
            );
            maskedEvent.setMDCPropertyMap(maskedMdc);
            
            // Replace event in appender chain (implementation-specific)
            // Note: Logback doesn't support event mutation easily
            // This is conceptual - actual implementation may require custom appender
        }
        
        return FilterReply.NEUTRAL;
    }
    
    private String maskSensitiveData(String message) {
        if (message == null) return null;
        
        // Mask patient IDs
        Matcher patientMatcher = PATIENT_ID_PATTERN.matcher(message);
        message = patientMatcher.replaceAll(match -> 
            SensitiveDataMasker.maskPatientId(match.group())
        );
        
        // Mask device serials
        Matcher deviceMatcher = DEVICE_SERIAL_PATTERN.matcher(message);
        message = deviceMatcher.replaceAll(match -> 
            SensitiveDataMasker.maskDeviceSerial(match.group())
        );
        
        return message;
    }
    
    private Map<String, String> maskMdcFields(Map<String, String> mdc) {
        if (mdc == null || mdc.isEmpty()) return mdc;
        
        Map<String, String> masked = new HashMap<>(mdc);
        
        // Check each MDC field for sensitive patterns
        for (Map.Entry<String, String> entry : mdc.entrySet()) {
            String value = entry.getValue();
            if (value != null) {
                String maskedValue = maskSensitiveData(value);
                if (!value.equals(maskedValue)) {
                    masked.put(entry.getKey(), maskedValue);
                }
            }
        }
        
        return masked;
    }
}
```

**Alternative simpler approach (recommended):**

Since Logback filters cannot easily mutate events, use a **custom appender wrapper** instead:

```java
public class SensitiveMaskingAppender extends UnsynchronizedAppenderBase<ILoggingEvent> {
    
    private Appender<ILoggingEvent> delegate;
    
    @Override
    protected void append(ILoggingEvent event) {
        // Create masked copy
        String maskedMessage = maskSensitiveData(event.getFormattedMessage());
        Map<String, String> maskedMdc = maskMdcFields(event.getMDCPropertyMap());
        
        LoggingEvent maskedEvent = new LoggingEvent(
            event.getLoggerName(),
            (LoggerContext) event.getLoggerContextVO(),
            event.getLevel(),
            maskedMessage,
            event.getThrowableProxy(),
            null
        );
        maskedEvent.setMDCPropertyMap(maskedMdc);
        
        // Delegate to actual appender (JSON console)
        delegate.doAppend(maskedEvent);
    }
    
    public void setDelegate(Appender<ILoggingEvent> delegate) {
        this.delegate = delegate;
    }
    
    // maskSensitiveData() and maskMdcFields() same as above
}
```

**Register in `logback-spring.xml`:**

```xml
<appender name="JSON_CONSOLE_MASKED" class="com.triagemate.triage.logging.SensitiveMaskingAppender">
    <delegate class="ch.qos.logback.core.ConsoleAppender">
        <encoder class="net.logstash.logback.encoder.LoggingEventCompositeJsonEncoder">
            <!-- ... encoder config -->
        </encoder>
    </delegate>
</appender>
```

**Why appender wrapper instead of filter:**
- Filters cannot mutate events (can only ACCEPT/DENY)
- DENY would drop critical ERROR logs containing sensitive data
- Appender wrapper rewrites event before JSON serialization
- Preserves all logs while masking sensitive content
- Checks both message AND MDC fields

    assertThat(logOutput).doesNotContain("DEV-SN-987654321");
    
    // Masked values present (deterministic hash)
    assertThat(logOutput).contains("PAT-"); // Prefix preserved
    assertThat(logOutput).contains("DEV-SN-"); // Prefix preserved
    
    // Validate hash format (8 hex chars after prefix)
    assertThat(logOutput).containsPattern("PAT-[0-9a-f]{8}");
    assertThat(logOutput).containsPattern("DEV-SN-[0-9a-f]{8}");
}

**Test implementation:**

```java
@SpringBootTest
@Testcontainers
@ActiveProfiles("docker") // CRITICAL: Activates JSON logging appender
class Phase11_1_StructuredLoggingTest {
    
    @Test
    void structuredLogging_shouldMeetAllCriteria() {
        // Given: Event with sensitive data
        InputReceivedEvent event = createEventWithSensitiveData();
        
        // When: Process event and capture logs
        LogCapture logCapture = LogCapture.start();
        kafkaTemplate.send("input-received", event);
        await().atMost(Duration.ofSeconds(5))
               .until(() -> logCapture.contains("decision executed"));
        logCapture.stop();
        
        // Then: Verify structured logging
        List<JsonNode> logs = logCapture.getLogsAsJson();
        
        // All logs are JSON
        assertThat(logs).isNotEmpty();
        
        // Business logs contain trace fields
        JsonNode decisionLog = findLog(logs, "decision executed");
        assertThat(decisionLog.get("requestId").asText()).isNotEmpty();
        assertThat(decisionLog.get("correlationId").asText()).isNotEmpty();
        assertThat(decisionLog.get("eventId").asText()).isEqualTo(event.eventId());
        assertThat(decisionLog.get("decisionOutcome").asText()).isIn("ACCEPTED", "REJECTED");
        
        // No sensitive data leaked
        String allLogs = logCapture.getAllLogsAsText();
        assertThat(allLogs).doesNotContain(event.patientId());
        assertThat(allLogs).doesNotContain(event.deviceSerialNo());
        
        // No noisy INFO logs
        long infoLogCount = logs.stream()
                                .filter(log -> log.get("level").asText().equals("INFO"))
                                .count();
        assertThat(infoLogCount).isLessThan(5); // Only business transitions
    }
}
```

**Why `@ActiveProfiles("docker")`:**
- Without it, test runs with default profile
- Logback activates plain text appender (`!docker,!prod`)
- `logCapture.getLogsAsJson()` would fail parsing plain text
- `docker` profile ensures JSON appender is active during test
- Alternative: Create dedicated `test-json` profile if needed
