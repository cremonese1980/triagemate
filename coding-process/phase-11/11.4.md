## Objective

Expose outbox-level metrics that capture publishing behavior, backlog health, and failure rates:

- **Publish counters:** Track successful publishes and retry attempts
- **Backlog gauges:** Monitor pending message count and oldest message age
- **Failure counters:** Track Kafka send failures and payload validation failures

These metrics enable backlog alerting, publisher health monitoring, and operational dashboards.

#### Technical Implementation

**Step 1: Create OutboxMetrics component**

**Create:** `com.triagemate.triage.metrics.OutboxMetrics`

```java
import io.micrometer.core.instrument.Counter;
import io.micrometer.core.instrument.MeterRegistry;
import org.springframework.stereotype.Component;

/**
 * Encapsulates outbox-related metrics.
 *
 * Counter metrics are eagerly registered to ensure they appear
 * in /actuator/prometheus even before the first event.
 */
@Component
public class OutboxMetrics {

    private final Counter publishedCounter;
    private final Counter retryCounter;
    private final Counter kafkaFailureCounter;
    private final Counter validationFailureCounter;

    public OutboxMetrics(MeterRegistry registry) {
        this.publishedCounter = Counter.builder("triagemate.outbox.published.total")
                .description("Total outbox messages successfully published to Kafka")
                .register(registry);

        this.retryCounter = Counter.builder("triagemate.outbox.retry.total")
                .description("Total retry attempts for failed outbox publishes")
                .register(registry);

        this.kafkaFailureCounter = Counter.builder("triagemate.kafka.publish.failure.total")
                .description("Total Kafka send failures during outbox publishing")
                .register(registry);

        this.validationFailureCounter = Counter.builder("triagemate.outbox.validation.failure.total")
                .description("Total payload validation failures during outbox processing")
                .register(registry);
    }

    /** Record a successful publish to Kafka. */
    public void recordPublished() {
        publishedCounter.increment();
    }

    /** Record a retry attempt after a failed publish. */
    public void recordRetry() {
        retryCounter.increment();
    }

    /** Record a Kafka send failure. */
    public void recordKafkaFailure() {
        kafkaFailureCounter.increment();
    }

    /** Record a payload validation failure. */
    public void recordValidationFailure() {
        validationFailureCounter.increment();
    }
}
```

**Why eager registration:**
- Counters appear on `/actuator/prometheus` immediately (value 0.0)
- No `MeterNotFoundException` in queries before first event
- Consistent with Micrometer best practices for counters

**Why no labels on these counters:**
- Each counter tracks a single concern — no outcome dimension needed
- Labels add cardinality without value here
- Simple, flat namespace is easier to query and alert on

**Step 2: Instrument OutboxPublisher**

**Update:** `com.triagemate.triage.outbox.OutboxPublisher`

```java
@Component
public class OutboxPublisher {

    private static final Logger log = LoggerFactory.getLogger(OutboxPublisher.class);

    private final JdbcOutboxRepository repository;
    private final KafkaTemplate<String, String> kafkaTemplate;
    private final OutboxProperties properties;
    private final OutboxMetrics metrics;
    private final String instanceId;

    public OutboxPublisher(
            JdbcOutboxRepository repository,
            @Qualifier("outboxKafkaTemplate")
            KafkaTemplate<String, String> kafkaTemplate,
            OutboxProperties properties,
            OutboxMetrics metrics
    ) {
        this.repository = repository;
        this.kafkaTemplate = kafkaTemplate;
        this.properties = properties;
        this.metrics = metrics;
        this.instanceId = resolveHostname() + "-" + UUID.randomUUID();
    }

    private static String resolveHostname() {
        try {
            return InetAddress.getLocalHost().getHostName();
        } catch (UnknownHostException e) {
            return "unknown";
        }
    }

    @Scheduled(fixedDelayString = "${triagemate.outbox.poll-interval-ms:2000}",
            initialDelayString = "${triagemate.outbox.initial-delay-ms:2000}")
    public void poll() {

        List<OutboxEvent> batch =
                repository.claimBatch(properties.getBatchSize(), instanceId);

        if (batch.isEmpty()) {
            return;
        }

        for (OutboxEvent event : batch) {
            publish(event);
        }
    }

    private void publish(OutboxEvent event) {

        try {
            kafkaTemplate.send(
                    event.getAggregateType(),
                    event.getAggregateId(),
                    event.getPayload()
            ).get();

            repository.markPublished(event.getId());
            metrics.recordPublished();

        } catch (Exception ex) {

            metrics.recordKafkaFailure();

            int attempts = event.getPublishAttempts() + 1;

            if (attempts >= properties.getMaxAttempts()) {

                log.error("Outbox exhausted for event {}", event.getId(), ex);
                repository.markExhausted(event.getId(), attempts, ex.getMessage());
                return;
            }

            metrics.recordRetry();

            long delay = computeBackoff(attempts);
            Instant nextAttempt = Instant.now().plusMillis(delay);

            log.warn(
                    "Outbox publish failed (attempt {}), retry in {} ms for event {}",
                    attempts, delay, event.getId()
            );

            repository.markFailed(
                    event.getId(),
                    attempts,
                    nextAttempt,
                    ex.getMessage()
            );
        }
    }

    private long computeBackoff(int attempts) {
        long base = properties.getBaseBackoffMillis();
        long exp = base * (1L << (attempts - 1));
        return Math.min(exp, properties.getMaxBackoffMillis());
    }
}
```

**Changes from existing code:**
- Inject `OutboxMetrics` via constructor
- `metrics.recordPublished()` after successful `markPublished()`
- `metrics.recordKafkaFailure()` on any exception (before retry/exhaust decision)
- `metrics.recordRetry()` only when a retry is scheduled (not on exhaust)

**Metric ordering rationale:**
- `recordKafkaFailure()` fires on every failure (including last attempt before exhaust)
- `recordRetry()` fires only when scheduling a retry (not on exhaust)
- This means: `kafka_failure_total >= retry_total` — the delta is exhausted events

### 11.4.b — Backlog Metrics

#### Goal

Expose real-time gauges for outbox backlog: pending message count and oldest message age.

These gauges are polled by Micrometer on each Prometheus scrape — no scheduled task needed.

---

#### Technical Implementation

**Step 1: Add query methods to JdbcOutboxRepository**

**Update:** `com.triagemate.triage.persistence.JdbcOutboxRepository`

```java
/**
 * Count pending outbox messages (status = PENDING).
 *
 * Used by OutboxMetrics gauge and health indicator (11.5).
 */
public long countPending() {
    String sql = """
        SELECT COUNT(*)
        FROM outbox_events
        WHERE status = ?
        """;

    Long count = jdbcTemplate.queryForObject(sql, Long.class, OutboxStatus.PENDING.name());
    return count != null ? count : 0L;
}

/**
 * Get the age in seconds of the oldest pending message.
 *
 * Returns 0 if no pending messages exist.
 *
 * Uses the existing idx_outbox_polling partial index
 * (status = 'PENDING') for efficient lookup.
 */
public double oldestPendingAgeSeconds() {
    String sql = """
        SELECT EXTRACT(EPOCH FROM (now() - MIN(created_at)))
        FROM outbox_events
        WHERE status = ?
        """;

    Double age = jdbcTemplate.queryForObject(sql, Double.class, OutboxStatus.PENDING.name());
    return age != null ? age : 0.0;
}
```

**Why these are on JdbcOutboxRepository:**
- Consistent with existing outbox query pattern
- Reuses the `idx_outbox_polling` partial index (`WHERE status = 'PENDING'`)
- Both methods are also useful for health indicator (11.5)

**Performance note:**
- `COUNT(*)` on partial index is fast (only PENDING rows indexed)
- `MIN(created_at)` uses the same partial index efficiently
- These queries run only on Prometheus scrape (default: every 15s)

```java
                .description("Total outbox messages successfully published to Kafka")
                .register(registry);

        this.retryCounter = Counter.builder("triagemate.outbox.retry.total")
                .description("Total retry attempts for failed outbox publishes")
                .register(registry);

        this.kafkaFailureCounter = Counter.builder("triagemate.kafka.publish.failure.total")
                .description("Total Kafka send failures during outbox publishing")
                .register(registry);

        this.validationFailureCounter = Counter.builder("triagemate.outbox.validation.failure.total")
                .description("Total payload validation failures during outbox processing")
                .register(registry);

        // Gauges (evaluated on each scrape — no scheduled task)
        Gauge.builder("triagemate.outbox.pending.count", repository, JdbcOutboxRepository::countPending)
                .description("Current number of unpublished messages in the outbox")
                .register(registry);

        Gauge.builder("triagemate.outbox.oldest.message.age.seconds", repository,
                        JdbcOutboxRepository::oldestPendingAgeSeconds)
                .description("Age in seconds of the oldest pending message")
                .register(registry);
    }

    // ... counter methods unchanged
}
```

**How Micrometer gauges work:**
- `Gauge.builder(name, obj, function)` registers a function pointer
- On each Prometheus scrape, Micrometer calls `repository.countPending()`
- No polling loop, no scheduled task, no stale data
- Value is always live at scrape time

**Why JdbcOutboxRepository injected into OutboxMetrics:**
- Gauges need a reference to the repository to call query methods
- Keeps gauge registration co-located with other outbox metrics
- Repository is already a Spring bean — no circular dependency risk

**Test implementation:**

```java
@SpringBootTest
@Testcontainers
@ActiveProfiles("test")
class OutboxMetricsLoadTest {

    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    @Autowired
    private MeterRegistry meterRegistry;

    @Test
    void outboxMetrics_shouldTrackPublishingBehavior() throws Exception {
        // Given - Capture baseline
        double baselinePublished = getCounterValue("triagemate.outbox.published.total");

        // When - Send 100 events
        for (int i = 0; i < 100; i++) {
            kafkaTemplate.send(
                    "triagemate.ingest.input-received.v1",
                    "evt-load-" + i,
                    buildValidEventPayload("evt-load-" + i)
            );
        }

        // Then - Wait for processing + publishing
        await().atMost(Duration.ofSeconds(60))
                .pollInterval(Duration.ofSeconds(2))
                .until(() -> getCounterValue("triagemate.outbox.published.total")
                        >= baselinePublished + 100);

        // Verify counters
        assertThat(getCounterValue("triagemate.outbox.published.total") - baselinePublished)
                .isEqualTo(100.0);

        // Verify backlog drained
        assertThat(getGaugeValue("triagemate.outbox.pending.count"))
                .isEqualTo(0.0);

        // Verify no failures
        assertThat(getCounterValue("triagemate.kafka.publish.failure.total"))
                .isEqualTo(0.0);
    }

    private double getCounterValue(String name) {
        try {
            return meterRegistry.get(name).counter().count();
        } catch (Exception e) {
            return 0.0;
        }
    }

    private double getGaugeValue(String name) {
        try {
            return meterRegistry.get(name).gauge().value();
        } catch (Exception e) {
            return 0.0;
        }
    }
}
```

**Why delta calculation:**
- Same rationale as 11.3: `meterRegistry.clear()` destroys all meters
- Delta approach preserves JVM, Hikari, and Spring Boot meters
- Safe for concurrent test execution

### Failure Injection — Kafka Unavailable

**Scenario:**
- Stop Kafka container
- Outbox publisher attempts to publish 10 pending messages

**Expected metrics:**

```prometheus
triagemate_kafka_publish_failure_total 10.0
triagemate_outbox_retry_total 10.0          # (if maxAttempts not yet reached)
triagemate_outbox_pending_count 10.0        # (messages still pending)
triagemate_outbox_oldest_message_age_seconds > 0
```

**Note:** Exact retry count depends on `maxAttempts` configuration. With `maxAttempts=3`, the first failure attempt produces 10 failures + 10 retries. On second poll (after backoff), another 10 failures + 10 retries. On third poll, 10 failures + 0 retries (exhausted).
