## Objective

Expose business-level metrics that capture decision processing behavior:

- **Decision counters:** Track outcomes (accepted, rejected, duplicate, invalid)
- **Decision latency:** Measure processing time with percentiles
- **Prometheus exposure:** Enable scraping via `/actuator/prometheus`

These metrics enable operational dashboards, alerting, and capacity planning.

**Step 3: Create metrics service**

**Create:** `com.triagemate.triage.metrics.DecisionMetrics`

```java
import io.micrometer.core.instrument.MeterRegistry;
import io.micrometer.core.instrument.Timer;
import org.springframework.stereotype.Component;

import java.time.Duration;
import java.util.function.Supplier;

/**
 * Encapsulates decision-related metrics.
 *
 * Uses dynamic tags - adding new outcomes requires no code changes.
 */
@Component
public class DecisionMetrics {

    private final MeterRegistry registry;

    public DecisionMetrics(MeterRegistry registry) {
        this.registry = registry;
    }

    /**
     * Record decision outcome and latency.
     *
     * @param outcome Decision outcome
     * @param latency Processing duration
     */
    public void recordDecision(DecisionOutcome outcome, Duration latency) {
        String tag = outcome.name().toLowerCase();

        // Increment counter
        registry.counter("triagemate.decision.total", "outcome", tag).increment();

        // Record latency
        registry.timer("triagemate.decision.latency.seconds", "outcome", tag)
                .record(latency);
    }

    /**
     * Record validation failure.
     */
    public void recordInvalid() {
        registry.counter("triagemate.decision.invalid.total").increment();
    }

    /**
     * Record decision error (exception during processing).
     */
    public void recordError() {
        registry.counter("triagemate.decision.error.total").increment();
    }

    /**
     * Execute decision logic with automatic timing.
     *
     * Records both outcome counter and latency histogram.
     * On exception, records error metric and re-throws.
     *
     * @param decisionLogic Decision execution logic
     * @return Decision result
     */
    public DecisionResult timeDecision(Supplier<DecisionResult> decisionLogic) {
        long startNanos = System.nanoTime();

        try {
            DecisionResult result = decisionLogic.get();

            // Record outcome + latency
            Duration duration = Duration.ofNanos(System.nanoTime() - startNanos);
            recordDecision(result.outcome(), duration);

            return result;

        } catch (Exception e) {
            // Record error metric
            recordError();
            throw e;
        }
    }
}
```

**Why dynamic tags:**
- Adding new outcome (DEFERRED, ERROR) requires no code changes
- Single method handles all outcomes
- Scales naturally with business logic evolution
- No field explosion (3 counters → 1 counter with tags)

**Why Supplier instead of Callable:**
- No checked exception handling needed
- Clean API (no try-catch wrapper required)
- RuntimeExceptions propagate naturally

**Step 4: Instrument DecisionService**

**Update:** `com.triagemate.triage.service.DecisionService`

```java
@Service
@Slf4j
public class DecisionService {
    
    private final DecisionRepository decisionRepository;
    private final OutboxRepository outboxRepository;
    private final OutboxEventBuilder outboxEventBuilder;
    private final DecisionMetrics metrics;
    
    public DecisionService(
            DecisionRepository decisionRepository,
            OutboxRepository outboxRepository,
            OutboxEventBuilder outboxEventBuilder,
            DecisionMetrics metrics) {
        this.decisionRepository = decisionRepository;
        this.outboxRepository = outboxRepository;
        this.outboxEventBuilder = outboxEventBuilder;
        this.metrics = metrics;
    }
    
    @Transactional
    public DecisionResult executeDecision(InputReceivedEvent event) {
        // Validate input
        ValidationResult validation = validateEvent(event);
        if (!validation.isValid()) {
            metrics.recordInvalid();
            log.warn("validation failed errors={}", validation.errors());
            throw new ValidationException(validation.errors());
        }
        
        // Execute decision with timing (handles outcome + latency metrics)
        DecisionResult result = metrics.timeDecision(() -> executeDecisionInternal(event));
        
        // Add outcome to MDC for logging
        MDC.put("decisionOutcome", result.outcome().name());
        
        log.info("decision executed outcome={}", result.outcome());
        
        // Queue for publishing
        String envelopeJson = outboxEventBuilder.buildEnvelope(
            "DECISION_EXECUTED",
            result
        );
        
        OutboxMessage outboxMessage = new OutboxMessage(
            "Decision",
            result.decisionId(),  // Use decisionId from result
            "DECISION_EXECUTED",
            envelopeJson
        );
        outboxRepository.save(outboxMessage);
        
        log.info("decision persisted and queued for publishing");
        
        return result;
    }
    
    /**
     * Internal decision execution (timed by metrics).
     * 
     * Returns DecisionResult with outcome and decisionId.
     */
    private DecisionResult executeDecisionInternal(InputReceivedEvent event) {
        // Check for duplicate
        if (isDuplicate(event)) {
            log.warn("duplicate event detected eventId={}", event.eventId());
            return DecisionResult.duplicate(event.eventId());
        }
        
        // Execute decision logic
        Decision decision = performDecisionLogic(event);
        Decision saved = decisionRepository.save(decision);
        
        // Return result with decision ID
        return DecisionResult.from(saved);
    }
    
    // ... other methods
}
```

**Changes from 11.3.a:**
- Single `metrics.timeDecision()` call replaces separate counter/timer recording
- `DecisionResult` must now include `decisionId()` field
- No explicit `recordDecisionMetric()` method needed
- Error handling automatic via `timeDecision()` exception catch

**DecisionResult update required:**

```java
public record DecisionResult(
    String decisionId,
    DecisionOutcome outcome,
    String reason
) {
    public static DecisionResult duplicate(String eventId) {
        return new DecisionResult(null, DecisionOutcome.DUPLICATE, "Duplicate event");
    }
    
    public static DecisionResult from(Decision decision) {
        return new DecisionResult(
            decision.getId(),
            decision.getOutcome(),
            decision.getReason()
        );
    }
}
```

### 11.3.b — Decision Latency Metrics

#### Goal

Measure decision processing time with histogram to track percentiles.

Enables SLA monitoring and performance regression detection.

**Critical:** Use histogram buckets only (not client-side percentiles) for multi-instance deployments. Histograms are aggregatable across instances using Prometheus `histogram_quantile()`.

---

#### Technical Implementation

**Note:** This task builds on 11.3.a. The `DecisionMetrics.timeDecision()` method already records latency - no additional code changes needed. This section configures histogram buckets and explains percentile calculation.

---

**Step 1: Verify histogram configuration**

The `application.yml` configuration from 11.3.a already enables histogram buckets:

```yaml
management:
  metrics:
    distribution:
      percentiles-histogram:
        triagemate.decision.latency.seconds: true
      slo:
        triagemate.decision.latency.seconds: 0.01,0.05,0.1,0.25,0.5,1.0,2.5,5.0,10.0
```

**Configuration breakdown:**
- `percentiles-histogram: true` - Enables histogram bucket generation
- `slo` - Defines bucket boundaries for service level objectives
- Buckets span subsecond (10ms) to slow (10s) operations

**Why histogram-only (no `publishPercentiles`):**
- Histograms use buckets aggregatable across instances
- Client-side percentiles (publishPercentiles) cannot aggregate
- Prometheus `histogram_quantile()` calculates server-side from buckets
- Production-grade for distributed systems

---

**Step 2: Understand how latency recording works**

From 11.3.a, the `DecisionMetrics.recordDecision()` method already records latency:

```java
public void recordDecision(DecisionOutcome outcome, Duration latency) {
    String tag = outcome.name().toLowerCase();
    
    // Increment counter
    registry.counter("triagemate.decision.total", "outcome", tag).increment();
    
    // Record latency (populates histogram buckets)
    registry.timer("triagemate.decision.latency.seconds", "outcome", tag)
            .record(latency);  // ← Histogram populated here
}
```

When `timeDecision()` calls `recordDecision()`, the timer automatically:
1. Increments count
2. Adds to sum
3. Distributes into histogram buckets

**No additional code required** - latency tracking integrated with counter tracking.

public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/actuator/prometheus").permitAll()
                .anyRequest().authenticated()
            );
        return http.build();
    }
}

**Test implementation:**

```java
@SpringBootTest
@Testcontainers
@ActiveProfiles("docker")
class DecisionMetricsLoadTest {
    
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    
    @Autowired
    private MeterRegistry meterRegistry;
    
    @Test
    void metrics_shouldTrackDecisionOutcomes() throws Exception {
        // Given - Capture baseline counts
        double baselineAccepted = getCounterValue("accepted");
        double baselineRejected = getCounterValue("rejected");
        double baselineDuplicate = getCounterValue("duplicate");
        
        // When - Send events with different outcomes
        sendEvents(70, "valid-accepted");   // Will be accepted
        sendEvents(20, "valid-rejected");   // Will be rejected
        sendEvents(10, "duplicate");        // Will be duplicate
        
        // Wait for processing
        await().atMost(Duration.ofSeconds(30))
               .until(() -> getCounterValue("accepted") >= baselineAccepted + 70);
        
        // Then - Verify counter deltas
        assertThat(getCounterValue("accepted") - baselineAccepted).isEqualTo(70.0);
        assertThat(getCounterValue("rejected") - baselineRejected).isEqualTo(20.0);
        assertThat(getCounterValue("duplicate") - baselineDuplicate).isEqualTo(10.0);
        
        // Verify latency recorded (check count deltas)
        assertThat(getTimerCount("accepted") - getTimerBaseline("accepted")).isEqualTo(70);
        assertThat(getTimerCount("rejected") - getTimerBaseline("rejected")).isEqualTo(20);
        assertThat(getTimerCount("duplicate") - getTimerBaseline("duplicate")).isEqualTo(10);
        
        // Verify latency is reasonable (< 1 second per decision)
        assertThat(getTimerMean("accepted")).isLessThan(1.0);
    }
    
    private double getCounterValue(String outcome) {
        try {
            return meterRegistry.get("triagemate.decision.total")
                .tag("outcome", outcome)
                .counter()
                .count();
        } catch (MeterNotFoundException e) {
            // Counter not yet created
            return 0.0;
        }
    }
    
    private long getTimerCount(String outcome) {
        try {
            return meterRegistry.get("triagemate.decision.latency.seconds")
                .tag("outcome", outcome)
                .timer()
                .count();
        } catch (MeterNotFoundException e) {
            return 0;
        }
    }
    
    private long getTimerBaseline(String outcome) {
        // Capture baseline before sending test events
        return getTimerCount(outcome);
    }
    
    private double getTimerMean(String outcome) {
        return meterRegistry.get("triagemate.decision.latency.seconds")
            .tag("outcome", outcome)
            .timer()
            .mean(TimeUnit.SECONDS);
    }
}
```

**Why delta calculation instead of clear():**
- `meterRegistry.clear()` removes ALL meters (JVM, Hikari, Spring Boot)
- Causes existing meter references to become orphaned
- Delta approach preserves all meters
- Safe for concurrent tests
